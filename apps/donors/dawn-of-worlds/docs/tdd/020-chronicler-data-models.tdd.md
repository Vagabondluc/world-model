## Acceptance Criteria

### AC-001: JournalEntry Data Model
**Given** a world event has occurred and been chronicled
**When** a JournalEntry is created
**Then** the entry must contain all required fields and be immutable

### AC-002: EntryType Classification
**Given** a journal entry is being created
**When** the entry type is specified
**Then** the type must be one of CHRONICLE, MYTH, or OBSERVATION

### AC-003: EntryScope Visibility
**Given** a journal entry is being created
**When** the scope is specified
**Then** the scope must be GLOBAL, REGIONAL, or LOCAL

### AC-004: Author Assignment
**Given** a journal entry is being created
**When** the author is assigned
**Then** the author must follow the event-type-to-author mapping rules

### AC-005: LoreTemplate Structure
**Given** a LoreTemplate is defined
**When** the template is used to generate an entry
**Then** the template must contain all required fields and valid generators

### AC-006: LoreContext Resolution
**Given** a world event and state
**When** context is being built
**Then** all canonical facts must be extracted and context variables must be valid

### AC-007: ChronicleCandidate Creation
**Given** a lore trigger has fired
**When** a ChronicleCandidate is created
**Then** the candidate must contain all required fields and have a valid status

### AC-008: ChroniclerSession Management
**Given** a chronicling session is started
**When** the session progresses through states
**Then** the session must follow the valid state transition rules

### AC-009: AutoChroniclerConfig Validation
**Given** an AutoChroniclerConfig is provided
**When** the config is validated
**Then** all numeric values must be within valid ranges

### AC-010: Entry ID Generation
**Given** a new journal entry is being created
**When** the ID is generated
**Then** the ID must follow the format je_A{age}_{sequence}

### AC-011: Myth Conflict Resolution
**Given** multiple myths exist for the same event
**When** myths are displayed
**Then** all variants must be shown with appropriate grouping

### AC-012: Entry Deletion Policy
**Given** a journal entry exists
**When** deletion is attempted
**Then** CHRONICLE entries must be append-only, OBSERVATION entries may be soft-deleted

## Testing Strategy

### Unit Testing Approach
- Test each data model type in isolation
- Validate type constraints and enum values
- Test ID generation algorithms
- Test context resolution logic
- Test state transitions for sessions and candidates

### Integration Testing Approach
- Test candidate creation from triggers
- Test template generation with context
- Test session lifecycle with backlog
- Test entry creation from sessions
- Test provenance tracking

### End-to-End Testing Approach
- Test full pipeline: event → trigger → candidate → session → entry
- Test backlog processing with auto-chronicler
- Test entry regeneration from event log
- Test myth conflict resolution in UI

### Performance Testing Approach
- Test context resolution with large world states
- Test template generation throughput
- Test backlog management with many candidates
- Test entry lookup performance
