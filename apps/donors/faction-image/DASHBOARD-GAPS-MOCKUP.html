🔍 Gap Analysis & Solutions

11
Shared State & Persistence
CRITICAL – Production Impact
Both entries share localStorage/sessionStorage & React Query cache → collisions
Namespace all keys and cache queries
Implementation:
// src/lib/storageKeys.ts export const STORAGE_KEYS = { app: { sidebarWidth: 'app_sidebarWidth', sectionCollapsed: 'app_sectionCollapsed', }, dashboard: { sidebarWidth: 'dashboard_sidebarWidth', readOnlyMode: 'dashboard_readOnlyMode', } }; // Use in React Query const QUERY_KEYS = { app: { icons: ['app', 'icons'] }, dashboard: { icons: ['dashboard', 'icons'] } };
Test: Verify localStorage in one entry doesn't corrupt the other.
12
Error Handling & Fallbacks
CRITICAL – Silent Failures
Dashboard crashes cascade; no graceful fallbacks or recovery UI
Error boundaries + entry-aware logging
Implementation:
// src/dashboard/DashboardErrorBoundary.tsx export class DashboardErrorBoundary extends React.Component { componentDidCatch(error, info) { logError({ entry: 'dashboard', error, timestamp: new Date() }); } render() { if (this.state.hasError) { return <DashboardErrorFallback onReload={() => window.reload()} />; } return this.props.children; } }
Add "Reload" and "Go to Main App" buttons in fallback UI.
13
Performance & Bundling
HIGH – Cost & UX
No bundle size targets; multi-entry bloat risk
Manualchunks, size monitoring, load budgets
vite.config.ts:
build: { rollupOptions: { output: { manualChunks: { 'shared-providers': ['src/app/AppProviders.tsx'], 'icon-engine': ['src/lib/iconRenderer.ts'] } } } } // CI check: error if bundle > 300kB
14
Security & CSP
CRITICAL – Compliance
No CSP headers; shared cookies; no session isolation
CSP meta tags, SameSite cookies, session key namespacing
In index.html & dashboard.html:
<meta http-equiv="Content-Security-Policy" content="default-src 'self'; script-src 'self' 'wasm-unsafe-eval';"> <script> // Domain-aware cookie setting document.cookie = 'session_id=...;SameSite=Strict;Secure'; </script>
15
Analytics & Feature Flags
HIGH – Product Insights
Can't measure dashboard adoption; no phase-gated features
Entry identifier + feature flag provider
In both main.tsx and main-dashboard.tsx:
window.__APP_ENTRY__ = 'main' | 'dashboard'; // Fire analytics with entry context trackEvent('icon_generated', { entry: window.__APP_ENTRY__, variantCount: 5 });
Add FeatureFlags provider; gate Phase 2+ features per entry.
16
Accessibility & Navigation
HIGH – WCAG Compliance
No keyboard shortcut mapping; no screen reader testing
Entry-aware keyboard hooks; axe-core automation
src/hooks/useKeyboardShortcuts.ts:
export function useKeyboardShortcuts() { if (window.__APP_ENTRY__ === 'main') { useBinding('Ctrl+G', generateIcon); } else if (window.__APP_ENTRY__ === 'dashboard') { // Bind Ctrl+G to no-op with tooltip } }
Include axe-core in CI; automate a11y audits.
17
Deployment & Versioning
HIGH – Ops & Rollback
No independent rollback; shared env var pollution
Entry-specific paths + versioning strategy
S3 paths:
s3://bucket/v1.5.0/index.html s3://bucket/v1.3.2/dashboard.html // .env.main and .env.dashboard kept separate VITE_API_BASE_URL=https://api.example.com VITE_ENTRY=main|dashboard
Document rollback: each entry can be reverted independently.
18
Testing Cleanup & Isolation
MEDIUM – Test Flakiness
Tests leak state; concurrent access not tested
Fixture setup; cache/storage clearing per test
Vitest beforeEach hook:
beforeEach(() => { localStorage.clear(); sessionStorage.clear(); queryClient.clear(); window.__APP_ENTRY__ = 'main'; });
Add concurrent access test: simulate both entries running simultaneously.
19
Developer Onboarding
MEDIUM – Velocity
No debugging guide; easy to break shared patterns
Documented patterns + lint rules
Guide: How to add a dashboard component:
  1. Create in src/dashboard/ComponentName.tsx
  2. Import only from src/dashboard/* & src/app/*
  3. If using router hooks → verify BrowserRouter in main-dashboard.tsx
  4. Pass CI linting (no imports from src/App.tsx)
20
Known Limitations & Future
MEDIUM – Roadmap
No roadmap for Phase 2+ features
Document opportunities & non-goals
Phase 2+ Opportunities:
  • Role-based access (viewer-only dashboard)
  • Offline mode caching
  • Dark theme override
  • Custom branding per deployment
Out of Scope: SSR, i18n, real-time collab, K8s scaling.

⚠️ Risk Assessment & Mitigation
🔴 Critical: State Collision

localStorage/sessionStorage keys overlap between entries

Mitigation: Namespace all keys with entry prefix; enforce in code review + linting

Test: Write integration test that modifies state in one entry, switches to other, verifies isolation

🔴 Critical: Silent Failure

Dashboard crashes with no recovery path; user left staring at blank page

Mitigation: Error boundary + fallback UI with "Reload" & "Go to Main" buttons; entry-aware error logging

Test: Inject error in provider; verify fallback renders

🟠 High: Security Regression

CSP/CORS policies not enforced at dashboard entry

Mitigation: Add CSP meta tags to both HTML files; test with CSP violations enabled

Test: Browser console CSP violation checks

🟠 High: Route Hook Coupling

Developer adds router hook to shared component; dashboard breaks

Mitigation: ESLint rule forbidding router hooks in shared code; auto-fix applies MemoryRouter wrapper if needed

Test: Lint rule pre-commit hook

🟠 High: Bundle Bloat

Multi-entry approach causes 2x bundling cost

Mitigation: Manual chunks for shared providers; size CI gate; target ≤300kB per entry

Test: CI reports bundle size deltas; fails if +15% quarter-over-quarter

🟠 High: Analytics Blindness

No way to track dashboard vs. main app usage; can't measure adoption

Mitigation: Set window.__APP_ENTRY__; include in all event payloads; dashboard dashboard-specific dashboard dimension in analytics

Test: Query analytics for "entry=dashboard" events

✅ Success Criteria
  • Dashboard endpoint loads without runtime errors
  • First paint DOM marker clearly identifies dashboard (not main app)
  • Shared providers are centralized; no duplication
  • Both entries can generate, export, and manage icons
  • State (localStorage, query cache) isolated by entry
  • Errors in one entry do not cascade to the other
  • Feature flags enable per-entry control
  • Bundle sizes tracked; no regression
  • All 11 gaps addressed per section 11–20 of spec
  • D1–D4 tests passing; CI gated
  • Developer guide published; team trained
